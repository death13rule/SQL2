# Домашнее задание к занятию "Репликация и масштабирование. Часть 2" - `Лютиков А.В.`

### Задание 1
Опишите основные преимущества использования масштабирования методами:

активный master-сервер и пассивный репликационный slave-сервер;
master-сервер и несколько slave-серверов;
Дайте ответ в свободной форме.

### Решение 1
Активный master-сервер и пассивный репликационный slave-сервер:
Преимущества:
1. Отказоустойчивость
2. Если master выходит из строя, slave можно быстро переключить в master
3. Резервирование данных
4. Все данные копируются на slave практически в реальном времени.
5. Минимизация downtime
6. При аварии время простоя сокращается.
7. Бэкапы без нагрузки на master
8. Резервные копии можно делать со slave.
9. Простота реализации
10. Самая простая схема репликации.
Недостаток:
1. Slave обычно простаивает или используется мало.

Master-сервер и несколько slave-серверов:
Преимущества:
1. Масштабирование чтения
2. SELECT-запросы распределяются между slave.
3. Разгрузка master
4. Master занимается только INSERT/UPDATE/DELETE.
5. Повышение производительности
6. Гибкость
7. Один slave можно использовать для аналитики, другой для резервного копирования.
8. Отказоустойчивость
9. При падении одного slave система продолжает работать.
Недостаток:
1. Запись ограничена одним master.


### Задание 2
Разработайте план для выполнения горизонтального и вертикального шаринга базы данных. База данных состоит из трёх таблиц:

пользователи,
книги,
магазины (столбцы произвольно).
Опишите принципы построения системы и их разграничение или разбивку между базами данных.

Пришлите блоксхему, где и что будет располагаться. Опишите, в каких режимах будут работать сервера.

### Решение 2
Исходные таблицы:
users
books
stores

1. Вертикальный шардинг

Разделяем таблицы по логике.

БД 1: UserDB
users(id, name, email, phone)

БД 2: BookDB
books(id, title, author, price)

БД 3: StoreDB
stores(id, name, city, address)

Преимущества разделения:
1. Изоляция сервисов;
2. Независимое масштабирование;
3. Проще поддержка;
4. Безопасность.
   
Схема:
```
                API
                 |
    ---------------------------
    |            |            |
    v            v            v
 UserDB       BookDB       StoreDB
 ```

2. Горизонтальный шардинг

Разбиваем данные внутри таблиц.

Например, таблица users:

Shard 1
id 1–1000

Shard 2
id 1000–2000

Shard 3
id 2000-...

Для books — по категории:

Shard 1
fiction

Shard 2
science

Shard 3 
education

Для stores — по городам:

Shard 1
Moscow

Shard 2
St.Petersburg

Shard 3
Khabarovsk

Схема:
```
                    Router
                      |
        --------------------------------
        |              |              |
        v              v              v
     Shard 1        Shard 2        Shard 3
```

Комбинированный вариант:

```
                      API Gateway
                           |
     ------------------------------------------------
     |                     |                        |
     v                     v                        v
  UserDB                 BookDB                  StoreDB
     |                     |                        |
  ---------           ----------               ----------
  |   |   |            |   |   |               |    |   |
 U1  U2  U3           B1  B2  B3               S1  S2   S3

```

Где:

U1/U2/U3 — шарды пользователей
B1/B2/B3 — шарды книг
S1/S2/S3 — шарды магазинов
